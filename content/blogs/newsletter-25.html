Thank You Alex

For those that don't know, Alex Ionescu, the kernel coordinator, has resigned.  Alex first joined the project in 2004, around the 0.2.2 release.  Since then, he's been at the center of quite a few squabbles about how to code the kernel.  However, Alex has also been responsible for completely rewriting the kernel almost from the ground up.  Today, about 60% of the kernel code is probably his.  The reason for Alex's departure is because of his joining David Solomon's Expert Seminars as an instructor.  Because this job would place him in close contact with many Microsoft programmers and also give him access to other Microsoft properties, continuing with the project would have resulted in possible conflicts of interest.  Alex has also stated that he feels he's done everything he can for ReactOS, as the components that still need to be completed are those that he has little knowledge of.  The ReactOS community thanks Alex for everything he has done and we wish him the best with his new career.

More details can be found on Alex's blog: http://www.alex-ionescu.com/?p=42

Catching Up

With the more or less completion of the kernel rewrite, the developers have decided to try and update the rest of the project accordingly.  Alex's work greatly increased completeness and compatibility, but unfortunately many other components didn't keep pace and didn't take advantage of new features made available.  As such, quite a good deal of user mode components have to be updated.  These include shell32, kernel32, gdi32, and user32.  Also, crss, the client server runtime system also needs work.  Now that the kernel will hopefully stabilize or at least not face any major overhauls anytime soon, the developers can breathe easy and do some work that might be more visible to end users.  An improved win32k subsystem would allow more applications to work, and not through hacks.

Infrastructure Updates

A few behind the scene changes have been taking place recently.  The first is an update of MediaWiki, finally fixing displaying images.  Also, RosCMS v3 has been undergoing more stress testing.  I'm not going to offer any date on when it'll be released, but it's getting there.  Ironing out the bugs on something so complicated takes time.  But as we get closer, the flaws and bugs with the current CMS have grown ever more apparent and those of you who've spoken to me about them, rest assured that every one I remembered to talk to Klemens about has been addressed in the update.

Preview versions of both the new ReactOS Build Environment and Test Environment have been floating around for a while now.  The update to the BE had a few teething issues but now seems to be fully functional.  A formal link to it will be put up later, as right now we're still experimenting with GCC 4.2.0.  However, if you wish to build trunk yourself, it is advised you get the BEs in this thread:

http://www.reactos.org/forum/viewtopic.php?t=3760&postdays=0&postorder=asc&start=0

or for UNIX/Linux:

http://www.reactos.org/forum/viewtopic.php?t=3922

The Test Environment is located here:

http://www.reactos.org/forum/viewtopic.php?t=3886

The TE is for Windows NT systems and includes a version of QEMU, KQEMU, a GUI for QEMU, and vdk, a virtual disk utility.  The current GUI will be familiar to some people, as it's the one Magnus started but abandoned.  As the GUI was written in C#, you'll need .NET to run it.  However, Ged Murphy, the developer and maintainer of the TE, is rewriting the GUI for QEMU.  His objective is to make it a rival to the VMware's interface and from what I've seen, he just might pull it off.  The rewrite is still in progress so we're stuck with the older GUI.  Ged says to give him a month, I say we don't hold him to it.  The rewritten GUI will also require .NET.
