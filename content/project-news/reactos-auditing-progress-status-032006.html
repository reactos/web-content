ReactOS Auditing Progress - Status 03/2006

A lot of things happened in january and february 2006. This news issue's goal is to sum up what has been happening in the past few weeks. If you want to stay tuned, visit the ReactOS Blogs from time to time. And do not forget to add the ReactOS News Feed to your feed reader or aggregator.
A lot of information of this news issue came from Ged Murphy's blog entry, thank you for clearing up of some questions.

ReactOS Auditing Frequently Asked Questions

If you have still questions after you read this FAQ then please visit the ReactOS Forum.

What is ReactOS? Is ReactOS based on Microsoft® Windows®? And so on ...

Please visit the ReactOS FAQ page for further information about ReactOS.

Why does the ReactOS developers audit their source code? Almost all other well known operating systems aren't (fully) audited either and are still in use.

In order for the project to stand the alpha-phase time, it must be 100% legal. Maybe some areas of the audit are a bit of an over reaction (more information), but they have to be addressed. Hiding or even ignoring things does not make them go away. The next time someone questions the legality of our code, we can "throw" the audit (documents) back at them.

Why has the audit progress bar been stuck on 15% for the past few weeks?

When the ReactOS developer team originally discovered that the ReactOS code base may contain code which had been reversed outside of US policy, we choose to carry out an audit by locking the current repository and opening a new one in which all audited code would be placed. The script which displays the audit status on the website was written based upon some methods in this repository and was set to update daily.
This repository is now dead (see voting), but the audit script is still running on that directory, hence, it’s been stuck at 15%. The old script will get replaced this weekend (see FAQ entry below).

Why is the outdated auditing progress bar still on the website frontpage? And what is the current auditing process.

The auditing process has changed a bit because it was realised by many of the devs that the original reaction had been a slight over-reaction. The only people raising questions about the methods of our reversing, came from the ReactOS developers (not from outside). Two votings later, the old repository is reopened for anonymous download with some new restrictions on the development side. The area which does effect is the development side. Changes cannot be made to files which are locked. In order to remove the lock, the code must pass an audit. The hope is that the audit will be carried out quicker via this method.

So where are we at now? - the current auditing status

The original repository is now back online and pretty much all the code is currently locked. The slow process of auditing and unlocking has now officially begun, however a big chunk of our code can bypass the audit.

When will be the next ReactOS release?

The ReactOS Development Team expect ReactOS 0.3 to be released soon. The new method we have taken will allow development to continue alongside the audit. Each consecutive release will have more audited code than the previous until we reach a certain release, where all code passed the audit.

What about the progress bar?

Ged Murphy wrote a new audit script for the new/old repository which is much more accurate than the previous one. This will be put into place this weekend.
